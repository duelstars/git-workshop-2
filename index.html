Git Commando's

Bestanden met code worden actief bewerkt en vervolgens op de computer (local machine) opgeslagen. Om geschreven code te verzekeren voor behoud, worden de bestanden ook op een server (remote of origin) opgeslagen. Het overdragen van die bestanden wordt via Git uitgevoerd. De bewerkte bestanden worden voor ieder project afzonderlijk naar Git overgedragen. Het tegelijkertijd overdragen van bestanden uit verschillende projecten is dus niet mogelijk.

Files van Local naar Remote:

  1. Stage bestanden met het commando: git add .
  2. Controleer met git status
  3. Commit bestanden met het commando: git commit -m "omschrijving van project wijzigingen"
  4. Contoleer met git status
  5. Push bestanden met het commando: git push origin main
  6. Controleer metgit status

Branches

Aftakkingen in tijd

Individueel project

Een developer dat zelfstandig aan een project werkt, gebruikt de main branch. Dat is de hoofd tijdlijn van het project.


Samenwerken aan een project

Het is mogelijk om bestanden die op de server (remote of origin) staan over te dragen naar een individuele computer (local). Bij het samenwerken aan een project kunnen meerdere developers bestanden van de Remote naar de eigen computer overdragen. Dit gaat middelsgit clone Iedere developer dat op hetzelfde moment op Github is ingelogd, is verbonden met de main branche (hoofd tijdlijn). Zodra een developer zelfstandig aan een opdracht werkt, maakt deze een feature branch (afzonderlijke branche) aan. Er ontstaat hierdoor een aftakking in de tijd. Zo kan elke developer aan een eigen opdracht in het project werken. En zo ontstaan er meerdere feature branches. Op elke branch zullen commits gemaakt worden. Zodra een developer klaar is met de opdracht kan die de feature branch naar de Remote overgedragen viagit push origin main

Het is conventie om de code in de main branch correct en foutloos te houden. Zodra er tevredenheid en consensus is over een individuele feature branch kan deze via git mergesamengevoegd worden met de main branch. Daarom wordt niet iedere feature branch zondermeer samengevoegd.

Zodra een feature branch naar de Remote is overgedragen, zal Github vragen of er een Pull Request uitgevoerd moet worden. De Pull Request is een overzicht van de veranderingen tussen de nieuwe code van de feature branche en de code van de main branch. De Pull Request toont het daadwerkelijke verschil tussen beide codes. Als de Pull Request geen resultaat toont is er dus geen verschil tussen beide codes.

Zodra de Pull Request is uitgevoerd kan de code in de feature branch middels een review door een andere (senior) developer gecontroleerd worden. Deze zal in de review feedback gegeven over de geschreven code. De Pull Request wordt dan met feedback voorzien. Na het opvolgen en voltooien van de aanbevelingen wordt de Pull Request opnieuw verzonden.

Zodra de code aan de gewenste eisen voldoet geeft de controleur akkoord middels Approve. De gewijzigde feature branch kan nu met de main branch worden samengevoegd. Gebruik in de Pull Request hiervoor de buttonMerge. Op de Remote staat nu de samengevoegde code. Op de local is deze samenvoeging nog niet zichtbaar, de tijdlijn op de local loopt achter op de tijdlijn van de Remote.

Het is (in uitzonderlijke gevallen) mogelijk om een Merge terug te draaien via een Rollback. Maar het is gebruikelijker om fouten op te lossen met het aanmaken van een nieuwe feature branch. Dit is eenHotFix.

Synchroniseren van Remote en local

Om de tijdlijnen van de local te synchroniseren met de tijdlijn van de Remote wordt git pull origin main gebruikt. De feature branch kan nu van de local verwijderd worden, aangezien die code nu in de main branch aanwezig is. Een andere developer die aan een eigen feature branch werkt en deze nog niet had samengevoegd met de main branch, loopt door een eerdere samenvoeging achter op de tijdlijn van de main branch. Voordat de developer diens feature branch gaat samenvoegen zal er eerst een git pull origin main uitgevoerd moeten worden.

Een developer kan nu met een nieuwe taak beginnen waardoor het voorgaande process zich weer zal herhalen.

Local Feature branch

Een nieuwe feature branch aanmaken op de local machine:

Voer in de terminal het commando git checkout -b naam-van-branch

Om te controleren hoeveel branches aangemaakt zijn, voer je het commando git branch in. Tevens wordt hier de branch waarin je werkt in groene tekst weergegeven.

Om te veranderen naar een andere branch, voer je het commando git checkout naam-van-branch in.

Om terug te gaan naar de main branch, voer je het commando git checkout main in.

Na het maken van wijzigingen volg je onderstaande stappen:

  1. git add .
  2. git commit -m "naam bericht"
  3. git push origin naam-van-branch

Met de laatste code wordt het bestand naar de Remote verstuurd.

Pull Request

Nu kan in de terminal de voorgestelde link gevolgd worden om een Pull Request aan te maken. In Github staat de Pull Request klaar, kies voor Create Pull Request.

Als een collega developer de code in de Pull Request wilt controleren, kan deze zijn aanbevelingen per regel code bij Files Changedtoevoegen. Zodra de aanbevelingen in de review klaar zijn wordt deze met submit review verzonden. Bij consensus over de code wordt middels Approve akkoord gegeven om de feature branch en de main branch samen te voegen middels Merge Pull Request. In Github kan de feature branch nu verwijderd worden aangezien feature- en de main branch nu gelijk zijn. Om nu de main branch naar de local over te dragen voer je het commando git pull origin main in.
